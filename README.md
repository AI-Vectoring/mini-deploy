# mini-deploy

A minimal Docker-based SSH server that gives you a reproducible, isolated environment accessible via SSH from your local machine. The goal is a clean, disposable remote shell — useful as a deployment target, a sandbox, or a base for further tooling.

---

## What it does

Builds a Docker container running an OpenSSH server, then wires up key-based SSH access so you can connect to it with a single command (`ssh mini-deploy`) from your host machine.

No passwords. No manual key copying. No persistent state between container restarts.

---

## How it works

### The pieces

| File | Role |
|------|------|
| `Dockerfile` | Defines the container image |
| `entrypoint.sh` | Runs inside the container at startup |
| `setup.sh` | Runs on the host — orchestrates everything |

---

### Step-by-step flow

#### 1. `setup.sh` — runs on the host

```
bash setup.sh
```

It does the following in order:

**a) Generates an SSH keypair** at `~/.ssh/mini-deploy` (ed25519) if one does not already exist. The public key goes to `~/.ssh/mini-deploy.pub`. This keypair is specific to this project.

**b) Builds the Docker image** tagged `mini-deploy-dev` from the local `Dockerfile`.

**c) Starts the container** named `mini-deploy-dev` with:
- Port `2222` on the host mapped to port `22` inside the container.
- `~/.ssh` from the host mounted read-only at `/home/appuser/.ssh-host` inside the container (available for future use).
- `~/.ssh/mini-deploy.pub` mounted read-only at `/tmp/authorized_keys` inside the container — this is how the public key is delivered to the container without baking it into the image.

**d) Writes an SSH config entry** to `~/.ssh/config`:

```
Host mini-deploy
  HostName localhost
  Port 2222
  User appuser
  IdentityFile ~/.ssh/mini-deploy
  StrictHostKeyChecking no
```

This is what allows `ssh mini-deploy` to work as a shorthand.

---

#### 2. `Dockerfile` — defines the image

Base image: `debian:13-slim`

Installs:
- `git`
- `openssh-client`
- `openssh-server`
- `ca-certificates`

Creates a non-root user `appuser` (UID 1000, GID 1000) with a home directory at `/home/appuser`.

Copies `entrypoint.sh` into the image and sets it as the container entrypoint.

Exposes port `22`.

---

#### 3. `entrypoint.sh` — runs inside the container at startup

```bash
mkdir -p /home/appuser/.ssh
cp /tmp/authorized_keys /home/appuser/.ssh/authorized_keys
chown -R appuser:appuser /home/appuser/.ssh
chmod 700 /home/appuser/.ssh
chmod 600 /home/appuser/.ssh/authorized_keys
exec /usr/sbin/sshd -D -e
```

- Copies the public key from `/tmp/authorized_keys` (mounted by `setup.sh`) into the correct location for SSH authorization.
- Sets the required filesystem permissions (`700` on `.ssh/`, `600` on `authorized_keys`) — sshd will reject keys if permissions are wrong.
- Starts `sshd` in the foreground (`-D`) with logging to stderr (`-e`), which keeps the container alive and makes logs visible via `docker logs`.

---

### Connection

After setup:

```bash
ssh mini-deploy
```

You land in a shell as `appuser` inside the container.

---

## Keys in play

There are three distinct sets of SSH keys involved:

### 1. sshd host keys — server identity
Location inside container: `/etc/ssh/ssh_host_{ecdsa,ed25519,rsa}_key`

Generated automatically when `openssh-server` is installed during `docker build`. These are the container's server identity — they prove to a connecting client that it is talking to the expected server. They are baked into the image and regenerated each time the image is rebuilt.

### 2. Project keypair — client authentication
Location on host: `~/.ssh/mini-deploy` and `~/.ssh/mini-deploy.pub`
Location inside container: `/home/appuser/.ssh/authorized_keys` (public key only)

Generated by `setup.sh` on first run. The private key stays on the host and is used by your SSH client to authenticate. The public key is mounted into the container at `/tmp/authorized_keys` and copied into place by `entrypoint.sh`. This keypair is what grants you access to the container.

### 3. Host `~/.ssh` mount — forwarded credentials
Location inside container: `/home/appuser/.ssh-host/` (read-only)

Your entire `~/.ssh/` directory on the host is mounted read-only inside the container. This gives `appuser` access to your host SSH keys (e.g. `github_key`, `aws`, `hetzner-cloud`) from within the container — so it can authenticate to external services like GitHub or remote servers without those keys being copied into the image.

---

## Key design decisions

**Client auth key is injected at runtime, not baked into the image.** The image contains no credentials. The public key is mounted as a volume at container start and copied into place by `entrypoint.sh`. This means the image is generic and reusable.

**Non-root user.** The SSH session runs as `appuser` (UID/GID 1000), not root. `sshd` itself starts as root (required to bind port 22 and read host keys), but drops to `appuser` for the session.

**No persistent volumes.** The container has no named volumes. Anything written inside the container is lost when the container is removed. This is intentional — the environment is disposable.

**`StrictHostKeyChecking no`** in the SSH config means the host key fingerprint is not verified. Acceptable for a local container; remove this if connecting over a real network.

---

## Requirements

- Docker
- `ssh-keygen` (standard on Linux/macOS)
- An existing `~/.ssh/` directory on the host

---

## Usage

```bash
# First time (or after removing the container)
bash setup.sh

# Connect
ssh mini-deploy

# Stop the container
docker stop mini-deploy-dev

# Remove the container
docker rm mini-deploy-dev

# Rebuild from scratch
docker rm -f mini-deploy-dev && bash setup.sh
```
